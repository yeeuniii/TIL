# 권한 관리 모델

## 1. RBAC (Role-Based Access Control, 역할 기반 접근 제어)

* 개념: 사용자의 '역할(Role)' 을 기반으로 시스템 접근 권한을 관리하는 모델. (NIST, 미국 국립표준기술연구소에서 발표)
* 구조 (Level 1. Flat RBAC): `Users(사용자)` ↔ `Roles(역할)` ↔ `Permissions(권한)`
* 특징: 데이터베이스 유지보수 시 역할(Role)을 부여하여 권한을 제어할 수 있음. 기존 통제 모델의 단점을 보완.

## 2. ABAC (Attribute-Based Access Control, 속성 기반 접근 제어)
* 개념: 접근하려는 주체가 가진 '특징(Attribute)' 을 기반으로 권한을 부여하는 모델.

## 3. 비교: 
내가 두 모델의 차이를 이해한 방식은 "무엇을 먼저 고려하는가?"이다. 두 모델은 인과관계와 접근 방식의 출발점이 다르다.

| 구분 | RBAC (역할 기반) | ABAC (속성 기반) |
| :--- | :--- | :--- |
| 출발점 | 사람의 역할 (Role) | 데이터의 규칙 (Policy/Rules) |
| 작동 방식 | 역할에 권한을 미리 세팅해 둠 | 접근하려는 순간 조건이 맞는지 검사함 |
| 성격 | 정적 (Static) | 동적 (Dynamic) |
| 예시 (마케팅팀) | 마케팅 관련 업무를 하는 '사람' 으로 묶어둔다. | 마케팅 관련 업무를 한다는 '조건' 을 확인한다. |

## 4. 실무 적용 고민 (사내 BI & 데이터 마트 환경)

현재 사용 중인 Metabase는 RBAC(그룹/컬렉션 기반) 체계로 만들어져 있다. 따라서 오픈소스 BI 환경과 전사 Raw Data(`fastlog`)가 오픈된 사내 문화를 고려했을 때, 다음과 같은 거버넌스와 데이터 마트 전략이 필요하다.

### ① Metabase 컬렉션 구조 분리 (공식 vs 샌드박스)
셀프서비스 BI를 지원하되, 데이터의 신뢰성을 지키기 위해 공간을 분리한다.
* Official (공식): 데이터팀만 수정 가능. 타 부서는 조회만 가능.
* Workspace (샌드박스): 특정 팀(예: 레벨디자인팀) 내에서 자체적으로 수정 가능. 필요시 타 부서에는 안 보이게 설정 가능.

### ② Raw 데이터 환경에서의 RBAC/ABAC 도입 의의
이미 전 직원이 `fastlog`에 접근할 수 있는 환경이라면, 권한 통제의 목적이 달라져야 한다.  
데이터를 못 보게 막는 것(보안)이 아니라, "누가 어디에 대시보드를 저장할 수 있는가"를 통제(관리)하는 것이 핵심이다. 무분별한 파생 대시보드 생성을 방지하고 책임 소재를 명확히 해야 한다.

### ③ 다음 분기를 위한 데이터 마트 설계 방향 (Action Item)
* 현재: 특정 대시보드(레벨 평가)에 필요한 필드만 최소한으로 파싱하여 중간 테이블 제공 중.
* 개선 (Phase 1 - 이번 분기): 현재 구축한 테이블들(`clearinfo`, `session`, `unclebill`)을 Metabase 모델로 띄워 레벨디자인팀에 PoC(테스트베드)로 제공. 현업의 데이터 활용 패턴 관찰.
* 개선 (Phase 2 - 다음 분기): 분석에 필수적인 '공통 차원(국가, OS, LTV 등)'은 넓게 포함시키고, 복잡한 JSON 페이로드(`v1~v15`)는 필요한 것만 타겟팅하여 파싱하는 주제별 OBT(One Big Table) 형태로 데이터 마트 고도화 진행 예정.
